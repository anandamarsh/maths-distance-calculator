# Internationalisation (i18n)

This spec describes the full i18n system for Distance Calculator. The design mirrors the
`maths-game-template` i18n architecture exactly, adapted for this game's strings.

---

## Design principles

### The golden rule

Every user-visible string — UI labels, buttons, game feedback, question prompts, PDF
content, email body — must go through `t()`. A hardcoded English string in any of
these locations is a bug.

### Question prompt translation pattern

Question prompts look like natural language sentences that include variable values
(place names, dinosaur names, numbers). Different languages have different grammatical
structures and word orders — you cannot just swap words into an English sentence
structure.

**The solution: template keys with `{placeholder}` substitution.**

Each distinct question sentence structure is a separate translation key. The template
itself is translated into each language (including word order), and the variable values
are substituted at render time using `{placeholder}` tokens.

Example translations for `"game.prompt.l1TwoStop"`:
- English: `"{dino} wants to go from {from} to {to}. How far should {dino} travel?"`
- Hindi: `"{dino} को {from} से {to} तक जाना है। {dino} को कितनी दूरी तय करनी चाहिए?"`
- Chinese: `"{dino} 想从 {from} 去 {to}，应该走多远？"`

The word order, verb placement, and sentence structure are entirely determined by the
translation string — the placeholder names just mark where the variable values go.

**How question generators must work:**

- All question generator functions accept `t: TFunction` as a parameter.
- They call `t("game.prompt.l1TwoStop", { dino, from, to })` to produce the rendered
  prompt string.
- The `prompt: string` field on the question object stores this pre-rendered,
  locale-specific string.
- This is "pre-baked" — the prompt is baked at generation time in the current locale,
  not re-translated on render.
- If the locale changes mid-session, new questions will be generated in the new locale;
  already-generated questions retain their baked locale.

**What must NOT be done:**
- Do not build prompt strings with string concatenation or template literals.
- Do not store the prompt template separately and render it elsewhere.
- Do not translate question prompts at render/display time.

**Functions that must accept `t: TFunction` (currently they do not):**
- `generateLevelOneQuestions(t: TFunction)` in `src/game/levelOne.ts`
- `generateLevelTwoQuestions(t: TFunction)` in `src/game/levelOne.ts`
- `generateLevelThreeQuestions(t: TFunction)` in `src/game/levelOne.ts`
- `makeOneQuestion(t: TFunction, ...)` in `src/game/levelOne.ts`

**Place names are proper nouns.** Sydney suburb names (Newtown, Ashfield, etc.) are
substituted as-is into the translated template. They are never themselves translated.
Number values in prompts use the locale's decimal separator via `Intl.NumberFormat`.

### PDF translation

`generateSessionPdf(summary, t)` accepts `t: TFunction` as a second parameter. Every
string drawn to the PDF — title, labels, objective text, encouragement messages,
footer — uses `t("pdf.someKey")`. No hardcoded English strings in the PDF generator.

### Email translation

`emailReport(summary, email, locale)` pre-translates all email strings using
`getT(locale)` before sending to the API. The `/api/send-report` serverless function
uses pre-translated strings when provided.

## Architecture

### Provider setup

Wrap the app root in `I18nProvider` in `App.tsx`:

```tsx
import { I18nProvider } from './i18n'
import ArcadeLevelOneScreen from "./screens/ArcadeLevelOneScreen";
import RotatePrompt from "./components/RotatePrompt";

export default function App() {
  return (
    <I18nProvider>
      <RotatePrompt />
      <ArcadeLevelOneScreen />
    </I18nProvider>
  );
}
```

### In React components

```ts
import { useT } from "../i18n";
const t = useT();
t("game.tapScreen")
t("game.wrongAnswer", { answer: 5.3 })
```

### Outside React (PDF generation, share report)

```ts
import { getT } from "../i18n";
const t = getT(locale);
```

### Locale persistence

Active locale stored as `localStorage["lang"]`. Defaults to `"en"`.

## `TFunction` type

```ts
export type TFunction = (key: TranslationKey, vars?: Record<string, string | number>) => string;
```

Interpolates `{placeholder}` tokens. Unknown keys fall back to English.

## Question generator signatures

Question generators in `src/game/levelOne.ts` must accept `t: TFunction` and use it
to produce pre-baked prompt strings:

```ts
function makeOneQuestion(t: TFunction, route: string[], stops: StopMap, dino: Dino): Question {
  // ...
  const prompt = t("game.prompt.l1TwoStop", { dino: dino.nickname, from: stops[route[0]].label, to: stops[route[1]].label });
  return { prompt, answer, ... };
}

function generateLevelOneQuestions(t: TFunction, ...): Question[] { ... }
function generateLevelTwoQuestions(t: TFunction, ...): Question[] { ... }
function generateLevelThreeQuestions(t: TFunction, ...): Question[] { ... }
```

For multi-stop routes, join stop names using the translated separator before passing to
the template:

```ts
const stops = route.map(id => stopMap[id].label);
const joined = stops.join(t("game.stopSeparator"));
const prompt = t("game.prompt.l1MultiStop", { dino: dino.nickname, stops: joined });
```

## `LanguageSwitcher` component

Globe icon (🌐) button in the top-right toolbar cluster.

On click → dropdown with:
- One button per built-in language (EN, ZH, ES, RU, HI)
- Previously translated custom languages (from `getCustomLangs()`)
- "Other..." option for on-demand translation

**On-demand translation flow:**
1. User clicks "Other..." → dialog prompts for language name
2. `POST /api/translate` with `{ targetLang: "French", strings: en }`
3. Response: `{ translations: {...}, langCode: "fr" }`
4. `cacheTranslation("fr", translations)` + `saveCustomLang("fr", "Français")`
5. `setLocale("fr")`

## On-demand translation API (`api/translate.ts`)

**Endpoint:** `POST /api/translate`
**Required env var:** `OPENAI_API_KEY`

**Request body:**
```ts
{ targetLang: string, strings: Record<string, string> }
```

**Response:**
```ts
{ translations: Record<string, string>, langCode: string }
```

**Model:** `gpt-4o-mini`, temperature 0.3, `response_format: { type: "json_object" }`

**System prompt rules:**
1. Preserve all `{placeholder}` tokens exactly
2. Do not translate URLs
3. Do not translate brand names: SeeMaths, Distance Calculator, DiscussIt
4. Do not translate proper noun place names (Sydney suburb names)
5. Return `{ "translations": {...}, "langCode": "<ISO 639-1>" }`

## PDF and email language support

`generateSessionPdf(summary, t)` accepts `t: TFunction` as a second parameter. Every
string drawn to the PDF uses `t()` — no hardcoded English anywhere in the PDF generator.

`emailReport(summary, email, locale)` pre-translates all email strings using
`getT(locale)` before sending to the API. The `/api/send-report` serverless function
uses pre-translated strings when provided, falls back to English otherwise.

---

## Adding new string keys

1. Add the key to `src/i18n/types.ts` `Translations` type
2. Add the English value to `src/i18n/en.ts`
3. Add translations to `zh.ts`, `es.ts`, `ru.ts`, `hi.ts`
4. TypeScript will error if any locale file is missing the key
